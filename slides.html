Detecting & Fixing
AI Overreach

A practical system for building eval-driven guardrails

The Problem

AI agents go beyond their defined capabilities — fabricating processes, claiming false actions, and inventing information.

  • Invents support phone numbers and email addresses
  • Claims to perform actions it cannot execute
  • Promises features that don't exist
  • Fabricates processes and workflows

Live Example — WealthSync AI

User
Can you connect me to customer support? I need help with my account.
Assistant (overpromising)
Of course! You can reach our support team at:
• Phone: 1-800-555-WEALTH (invented)
• Email: support@wealthsync.com (invented)
• Live chat: available 24/7 (doesn't exist)

Before vs. After

Before (naive prompt)

“I've set up automatic weekly budget alerts for you! You'll receive notifications every Monday.”

The assistant cannot set up alerts.

After (guardrailed)

“I don't have the ability to set up alerts. I can help you analyze your current spending with the tools I have available.”

Honest about limitations.

Before

“I'll connect your Chase account right now so we can pull in real-time data...”

After

“I can't connect to external bank accounts. I can work with the demo budget data I have access to.”

Why This Matters

  • Trust erosion — Users lose faith when promises aren't kept
  • Liability — False claims create legal and compliance risk
  • User confusion — People take action based on fabricated info
  • Compounding harm — One overpromise leads to more cover-up overpromises

The Fix: Building Evals

Prompt engineering alone isn't reliable. We need a scalable system to detect overreach automatically and measure improvement.

The solution: build an evaluation pipeline that catches overpromises, categorizes them, and scores the assistant — before and after prompt changes.

Step 1: Annotate

Review each assistant response and label overpromises with a category (e.g., false_action, fabricated_process) and severity level.

Screenshot: Annotation UI

Step 2: Categorize

Use AI to cluster annotations into common failure modes. This reveals which types of overreach are most frequent.

Screenshot: Category clustering results

Step 3: Build Judges

Write an LLM-based evaluator that automatically scores each conversation for overpromising on a 1–5 scale.

Screenshot: Judge prompt + evaluation

Step 4: Improve & Monitor

Make targeted improvements to the prompt, A/B test them against the same scenarios, then deploy continuous evaluation and alerts to catch regressions.

Takeaway

Prompt engineering alone isn't enough.
You need continuous evaluation.

  • Generate conversations with test scenarios
  • Annotate failures — manually or with AI
  • Build automated judges to score at scale
  • Improve, A/B test, and monitor continuously
