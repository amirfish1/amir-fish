Two Fixes for Running 30+ Claude Sessions in Parallel

Both have the same shape — turn implicit, hard-to-see state into legible, queryable artifacts.

I run dozens of Claude Code sessions in parallel daily to ship my projects. After months of doing this, two things were dragging the workflow to a crawl. Both have fixes now. This is what they are and how they work.

If you're running more than ten parallel sessions, you've probably hit at least one of these.

Problem 1

Sessions Fighting Over Commits

Five sessions on the same main branch, each wanting to commit their own work. They'd burn cycles checking git status, second-guessing whether files were theirs, sometimes clobbering each other anyway. The git index is a single shared file. Between the moment one session runs git add and the moment it runs git commit, a sibling session can stage their own work into the same index — and the commit happily picks up whatever is staged at commit time.

Pre-commit checks don't help: there's an unbounded race window between the check and the commit. I observed it bite twice in one day across two repos.

The fix — a shared markdown chat file

A GROUP_CHAT.md file lives in every working directory. All sessions read and write to it. Before committing, each session posts something like:

## 2026-05-04 — Proposal from `worktree-rebase-7c25`

Stakeholders: `archive-filter`, `pr-link-test`, `cwd-routing`.

### Proposed sequence
1. **`worktree-rebase-7c25`** — codex spawn engine
   - Stage: `git add server.py static/index.html`
   - Commit: `feat(ui): codex spawn — engine routing`
2. **Push:** `git push origin main` after step 1.

Sibling sessions reply with ✅ ack or Counter:. The first session to start posts ▶ starting step 1. After committing, it posts ✅ step 1 done. The protocol is just a markdown convention, enforced by a /group-chat slash-command skill that handles the role logic (proposer / acker / executor / observer).

Three things make it work:

  1. Stage by explicit path only. Never git add -A — it'll sweep in another session's in-flight edits.
  2. Commit with git commit --only <paths>. Atomic. The race window collapses to zero — only the listed paths land in the commit, regardless of what else is staged.
  3. Make coordination visible. The point isn't the file. It's that sessions can't accidentally race over invisible state.

No daemon, no IPC, no central server. Just a markdown file and a convention. Multiple sessions on the same Mac, on the same clone, on the same branch — they negotiate, they don't clobber.

Problem 2

Retrieving Across Sessions

I work on multiple products. The same topics surface across many sessions. "What did I decide about the customization-layer thesis last week?" used to take minutes of grep over ~/.claude/projects/*.jsonl. With dozens of session files spread across multiple projects, even ripgrep couldn't keep that fast.

That delay broke flow. By the time I'd found the relevant earlier conversation, the current session had already moved on or made the same mistake.

The fix — a local search index over every transcript

claude-index is a SQLite-backed index that ingests every Claude Code transcript on the machine. Two flavors of search live on top:

  • Keyword (FTS5). For exact strings — file paths, error messages, commit hashes, slugs. Fast, deterministic, perfect when you know the words.
  • Semantic (vector embeddings via Ollama). For "find that thing about hyper-personalization" when you don't remember the exact words. Hybrid Reciprocal Rank Fusion combines both signals.

Both run locally. Sub-second results across thousands of session files. The query language supports phrase quoting, prefix wildcards, time windows, project filters, and role filters.

Wired into CCC's sidebar — type a topic, matches surface live without opening a terminal:

// inside CCC: type "tax prep hub" in the search bar
// → returns the 3 sessions where you discussed it,
//   ranked by recency × relevance, with snippets.

The index also covers Codex transcripts (~/.codex/sessions/*.jsonl) under the same schema — one search, both runtimes.

What changed in practice: retrieval went from "I'll grep this later" to "I just looked it up." That shift collapsed the cost of cross-session memory enough that I started leaning on it for decisions I used to make from gut.

Common Thread

Operations, Not Magic

Both fixes do the same thing: turn implicit, hard-to-see state into legible, queryable artifacts. Sessions chatting in markdown. Memory in a SQLite index. Nothing fancy.

When you're orchestrating dozens of agents, the real wins aren't smarter prompts or better models. They're better record-keeping. The implicit, ambient knowledge an individual developer carries in their head doesn't scale to thirty parallel sessions — but markdown files and search indexes do.

If you're running more than ten parallel Claude Code sessions, both of these will save you time. If you're running fewer, file them away — at some point the overhead of coordination starts to dominate the cost of work, and you'll want them ready.
